目标：看到一个 eval / rollout / calibration 脚本时，能拆出它的输入、流程、指标、决策边界。

1. 今日核心模型

一个 harness = 输入 + 固定流程 + 指标 + 决策边界。

只要你能拆出这四件事，就基本能判断这个实验结果到底能证明什么、不能证明什么。

2. 四件套分别是什么

部件要问的问题本项目里的常见形态
输入 input它吃什么数据？数据是否固定？replay sqlite、case json、self-play corpus、EV label queue、profile 文件
固定流程 procedure它每次是否按同样步骤跑？固定 seed、固定 max_turns、固定 profile、固定参数网格、固定候选动作
指标 metrics它用什么数字判断好坏？accuracy、holdout_accuracy、EV delta、positive_rate、changed-choice、trigger rate
决策边界 decision boundary结果能导致什么动作？不能导致什么动作？reject、diagnostic_only、needs_label、needs_shadow、candidate_review_ready、manual profile gate

3. 用真实脚本拆一次

看 scripts/evaluate_replay_calibration.py。

四件套脚本里的对应物
输入DEFAULT_DATASET = data/tuning/replay_policy_calibration_dataset_20260601.json
固定流程_grid_hu() 和 _grid_discard() 对参数网格逐项评估；_hu_shadow_pred()、_discard_shadow_pred() 生成 shadow prediction。
指标accuracy、holdout_accuracy、weighted_margin、complexity、promotion_safe
决策边界报告明确说这些是 shadow evaluations，不是 automatic production changes。

4. 为什么“决策边界”最重要

很多 AI 调参误判，不是因为数字算错，而是因为把证据级别用错。

一个 harness 结果正向，不等于可以写 profile。

它可能只说明：这个方向值得继续 label、值得做 shadow、值得做 changed-choice audit，或者只值得保留为 diagnostic。

你们刚接入的十问 gate，本质上就是把“决策边界”机器化：

single_point        -> diagnostic_only
error_family no EV  -> needs_label
stable EV no shadow -> needs_shadow
safe all gates      -> candidate_review_ready

6. 今天你要记住的一句话

读 harness 先不要问“结果是不是正的”，
先问：输入是什么、流程是什么、指标是什么、结果允许我们做什么。

7. 课后项目练习

下次你看到一个训练/评估报告时，按下面模板写 4 行：

Input:
Procedure:
Metrics:
Decision boundary:

如果其中任何一行写不出来，就说明这个结果还不能被你放心使用。
