1. 今天的核心问题

同一个局面，强行走两个分支，重复模拟未来，比较长期收益。

这就是 replay / branch eval harness 的核心。它不是问“这一次实际后来发生了什么”，而是问：

如果当时 A：pass，会怎样？
如果当时 B：peng/chi/discard，会怎样？
在许多随机后续牌墙下，哪个长期平均更好？

2. 为什么不能只看单局结果

麻将有随机摸牌、多人行动、不完全信息。单局后续结果可能只是运气。

单局复盘误区Branch eval 的做法
“我这次过胡后来摸到了好牌，所以过胡一定对。”固定当前局面，重复模拟很多次，比较平均分。
“这次 peng 后放炮了，所以 peng 一定错。”比较 pass 分支和 peng 分支在同类后续随机下的长期 EV。
“AI 这手看起来很怪。”用反事实分支验证它是噪声、低触发，还是可推广错误。

3. 本项目里的真实 harness

看 scripts/training/run_claim_counterfactual.py。它的文件说明是：

Focused counterfactual: simulate pass vs human's actual peng/chi on selected cases.
Harness 四件套这个脚本里的对应物
Input--input 的 cases，包含 round_id、source_event_index、seat、human_action、tile
Procedure从 replay 重建局面；强制 pass 分支；强制 claim 分支；分别跑 simulate_round()
Metricspass_avg、claim_avg、delta = claim - pass、胜率差、pos/neg/tie
Decision boundary只产生 EV 证据；不能自动写 profile；正式 claim/pass 证据必须用 max_turns=180

4. 你要看懂的三个数字

数字意思怎么看
delta = claim - passclaim 分支平均分减 pass 分支平均分正数支持 claim，负数支持 pass
pos/neg/tie多个 seed 或多个 case 中，方向支持/反对/接近持平的数量比单个平均值更能看稳定性
max_turns=180模拟到足够长的未来，不用短截尾糊弄结论claim 分支收益报告少了它，只能当工具调试

本项目规则：claim 分支收益模拟禁止用短截尾 max_turns=80 当结论。正式 pass/peng/chi A/B 报告必须明确 max_turns=180。

5. Branch eval 的决策边界

一个 branch eval 正向，一般只说明“这个方向值得继续”，不等于能上线。

单个 case 正向
  -> 可能只是单点正例，diagnostic_only
多个同桶 case 正向
  -> 可能 needs_label / needs_shadow
changed-choice 足够 + EV 稳定 + A/B 安全
  -> 才可能 candidate_review_ready
人工批准
  -> 才能写 train/play profile

这正好对应你们现在接入的十问 gate。
