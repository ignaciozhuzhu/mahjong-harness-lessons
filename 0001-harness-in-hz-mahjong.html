1. 先给一个项目化定义

Harness = 可复现的实验外壳。
它把“候选策略/规则/模型”接到固定输入、固定流程、固定指标上，然后产出可比较的结果。

在本项目里，它经常不是线上 AI 决策本体，而是用来回答这类问题：

  • 这个 hu/pass 阈值要不要放宽？
  • 某个 discard 规则改了以后，holdout 会不会变差？
  • 一个 profile nudge 是否有足够 changed-choice 覆盖，值得正式评估？

2. 本仓库里的真实例子

最清楚的例子是 scripts/evaluate_replay_calibration.py。它的注释明确说：这不是最终模型，而是一个 reproducible harness，用来衡量未来标签是否支持扩大/收窄 guard。

harness 部件这个脚本里的对应物
固定输入data/tuning/replay_policy_calibration_dataset_20260601.json
候选规则_hu_shadow_pred()、_discard_shadow_pred()
参数搜索_grid_hu()、_grid_discard()
指标accuracy、holdout_accuracy、weighted_margin、complexity、promotion_safe
输出data/tuning/replay_policy_calibration_eval_20260601.md
结论边界报告里写明：shadow evaluations，不是 automatic production changes

3. 一个简单判断法：四问

  1. 输入固定吗？ dataset、seed、profile、replay 是否明确？
  2. 流程固定吗？ 是否每次按同样步骤运行？
  3. 指标固定吗？ 是否输出 accuracy / holdout / 胜率 / changed-choice 等？
  4. 边界清楚吗？ 是否说明结果只是诊断、候选证据，还是能直接改生产？

四个问题都能答出来，就很像 harness。

4. 和“生产代码”的区别

生产策略Harness
真实对局时直接决定打哪张/胡不胡离线复跑或模拟，用来测候选策略
目标是稳定服务玩家/AI目标是生成证据，帮助调参或否决方案
一次错误会影响实际行为一次错误通常影响实验结论，需要修 harness 或重跑
关注实时性和兼容性关注可复现、可比较、可解释

6. 你在本项目里应该怎么用这个词

建议你以后读报告时这样说：

这个 harness 的输入是什么？
它测的是生产策略，还是 shadow rule？
它有没有 holdout？
它的结论能不能 promotion，还是只能当 diagnostic？

这会避免一个常见误区：看到 harness 结果正向，就误以为可以直接上线。当前项目的训练规则里其实很强调：小样本、单 seed、低 changed-choice 覆盖，通常只能作为诊断。
